회의록 — 엔진/전투 조작/타겟 플랫폼

안건

엔진 선택, 그래픽 스타일(사용자가 이미 스타듀밸리 스타일로 지정 — 논쟁 대상 아님), 전투 조작 방식, 타겟 플랫폼/유저층에 대한 종합적 의견 도출. 2라운드로 진행: 1라운드(엔진/전투 큰 틀/플랫폼) → 사용자가 가디언테일즈·이상한 던전 시리즈 레퍼런스를 추가 제시 → 2라운드(액션전투 세부/절차적 던전/실패 안전판 재검토).

선행 확인 사항

엔진 추천은 팀 규모/코딩 경험에 좌우되므로, 리서치 전에 사용자에게 먼저 확인: 1인 개발, 코딩 가능. (이 답변으로 docs/roadmap.html의 "팀 규모" 미확정 항목도 함께 해소됨.)

참가

  • technical-advisor — 엔진/구현/플랫폼
  • game-designer — 철학 정합성
  • systems-designer — 전투/던전 메커닉 구체화
  • scope-reviewer — 1인 개발 스코프 비관적 검토

리서치 브리핑 (웹 조사, 전부 WebSearch 결과 종합 — 개별 원문 WebFetch는 안 함, 신뢰도는 각 항목에 표시)

1라운드

  • 스타듀밸리 기술 스펙: MonoGame/XNA 기반, 16x16 픽셀 타일이 기본 단위(스프라이트는 배수 확장). "스타듀 스타일"은 그림체 문제이지 엔진 종속 아님. 신뢰도: 중간(포럼/커뮤니티 소스). 출처: forums.stardewvalley.net, community.monogame.net
  • Godot vs Unity(2026): Godot는 2D 전용 파이프라인, MIT 라이선스(로열티 없음), 1인 개발 반복속도 유리. 콘솔 익스포트는 Unity가 더 직접적 경로이나, Godot도 공식 콘솔 익스포트 템플릿 존재 — 실제 콘솔 정식 출시엔 서드파티 포팅 비용 $10K~50K가 일반적. 신뢰도: 중간(블로그/기술매체 종합). 출처: dev.to, tech-insider.org
  • Coral Island(언리얼)/Fields of Mistria(엔진 미확인): 참고 수준.
  • Fields of Mistria 전투: 실시간 액션 — 검 공격, 회피, 콤보, 점프 회피, 강공격, 마법 시스템. "스타듀보다 전투 깊이가 낫다"는 평가. 신뢰도: 중간. 출처: fieldsofmistria.wiki.gg/wiki/Combat, wikipedia
  • 장르/플랫폼 동향: Steam "cozy" 태그 2020 대비 675% 성장. Fields of Mistria는 PC 얼리액세스 선출시 → 스위치2 포트 후속 발표, 이 장르의 일반적 패턴. 신뢰도: 중간. 출처: pcgamer.com, switchbladegaming.com

2라운드

  • 가디언테일즈 조작: 기본 공격(쿨다운 없음)+대시+무기 고유 스킬(캐릭터가 아닌 장착 무기에 종속)+조건부 체인 스킬. 신뢰도: 중간. 출처: guardian-tales.fandom.com
  • 이상한 던전 시리즈: 절차적 생성 던전 + 완전 영구사망이 아닌 "죽으면 마을로 리셋되지만 보관함 아이템은 유지"되는 완충형 로그라이크. 춘소프트가 원조 로그라이크를 "이해하기 쉽고 플레이하기 쉽게" 만들려는 의도로 설계. 신뢰도: 중간. 출처: wikipedia.org/wiki/Roguelike, litrpgreads.com

technical-advisorGodot 추천(조건부: 프리메이드 팜시뮬 에셋 대량구매 계획 있으면 Unity). 단순 실시간 액션 지지. PC 선출시.Godot 유지. 무기별 스킬은 구현 난이도 자체보다 콘텐츠 볼륨이 진짜 비용 — 다만 "특산물 가공 장비" 루프와 정합적. 절차적 생성은 보스룸 제외, 여정 구간만. 실패 리셋은 game-designer/systems-designer가 풀 문제로 명시(자기 판단 보류).
game-designer전투는 공격+회피 2버튼만, 모험 비중 확대는 콘텐츠 볼륨/준비단계 깊이로.가디언테일즈식 "기본공격만으로 진행 가능, 스킬은 있으면 더 강함"은 이미 확정된 철학 2번의 반복이라 새 리스크 아님 — 버튼 개수보다 "실행숙련 강요 여부"가 관건. 이상한 던전식 리셋+부분손실은 노숙 안전망과 정면 경합 + 철학 1번(농장→모험 상호의존)과 충돌 지적, 별도 옵트인 콘텐츠로 분리하는 절충안 제시.
systems-designer공격/회피/특수(장비반응) 3버튼 제안 — game-designer와 이견.무기 "아키타입"(3~4종 상한)으로 스킬 로직 공유 + 특산물 속성 파라미터 주입안 제시(조합폭발 완화). 이상한 던전 안전판을 실시간판으로 구체화(모닥불=은행 체크포인트, 미은행 전리품+층진행도만 손실)했으나 이는 사용자가 최종 반려.
scope-reviewerMVP 전투는 "휘두르기1종+회피+아이템사용"만, 콘솔 동시출시 반대, PC선출시 지지.무기별 스킬은 조합폭발 우려 재확인, 아키타입 1~2종만 MVP 권고. 절차적 생성은 "비용 절감이 아니라 비용 치환"이라며 M3로 유보 권고. 이상한 던전 리셋은 노숙 안전망과 경합한다며 반려 의견.

충돌 지점 (숨기지 않고 기록)

  • 2버튼(game-designer) vs 3버튼(systems-designer): 가디언테일즈 레퍼런스로 실질적으로 해소됨 — 관건은 버튼 개수가 아니라 "3번째 입력이 실행 숙련을 강요하는지, 장비로 얻는 옵션인지"라는 game-designer의 재정의에 systems-designer의 "무기 아키타입"안이 부합하는 형태로 수렴.
  • 절차적 던전 생성의 가치: technical-advisor/systems-designer는 "여정 구간 한정 절차 생성"에 긍정적이었으나, scope-reviewer가 "던전 개수가 적은 MVP 단계에서는 생성기 자체의 튜닝 비용이 핸드메이드보다 클 수 있다"고 반박 — 이 반박이 최종 채택되어 M3로 유보.
  • 이상한 던전식 실패 리셋 — 가장 첨예한 지점: systems-designer가 구체적 메커닉(모닥불=은행 체크포인트)까지 설계했으나, technical-advisor는 이미 확정된 노숙 안전망과의 경합 가능성을 지적하며 판단을 유보했고, game-designer와 scope-reviewer가 각각 철학적/스코프 근거로 반대 의견을 냄. 사용자가 최종적으로 완전 반려 — "플레이어의 의지를 꺾는 행위"라는 확고한 근거.

사용자 최종 결정 (2026-07-03)

  1. 엔진: Godot 확정.
  2. 타겟 플랫폼: Steam(PC) 출시를 우선 목표. 스위치 이식 가능성은 낮음.
  3. 전투 조작: 기본 2버튼(공격/회피) + 무기 아키타입(3~4종 상한) 종속 특수버튼 확정. MVP는 아키타입 1~2종만 우선 구현. 스타듀밸리도 WASD+마우스로 전투가 가능하다는 점을 참고해 키보드+마우스로 전체 플레이가 완결되어야 함. 마을/던전 세부 조작(이동, 카메라 등)은 추가 논의로 보류.
  4. 절차적 던전 생성: MVP에서는 하지 않음(순수 핸드메이드), M3 이후 재검토.
  5. 도전용/엔드리스 던전(별도 옵트인 콘텐츠): 우선사항 아님 — 백로그로 미룸(거부는 아님).
  6. 이상한 던전식 "던전 실패 시 마을 리셋 + 부분 손실" 안전판: 절대 채택하지 않음. 스토리/메인 던전은 물론 어떤 형태의 옵트인 콘텐츠로도 적용하지 않는다. 사용자 근거: "플레이어의 의지를 꺾는 행위".
