# 퀘스트 설계 스킬

**목적:** 개별 퀘스트를 즉흥적으로 산문으로 적지 않고, 트리거/목표/보상/실패조건이 빠짐없이 정의되도록 표준 구조로 `docs/quests/<퀘스트명>.md`에 정리한다.

**활성화:** "퀘스트 설계", "이 이벤트를 퀘스트로", "대리 운영 트리거 퀘스트" 등.

**표준 구조:**
```markdown
---
status: draft
village: <관련 마을 또는 공통>
type: 메인 | 서브 | 대리운영 트리거 | 노숙 이벤트
---

# <퀘스트명>

## 트리거
(어떤 조건/대화/장소 도달 시 발생하는지)

## 목표
(플레이어가 해야 하는 것)

## 보상
- 마을 한정 재화 여부 (docs/systems/settlement.html 참고)
- 특산물/아이템/호감도(옵트인 트리거인 경우만) 등

## 실패/포기 조건
(강제 콘텐츠가 아니어야 함 — CLAUDE.md 철학 3번 옵트인 원칙 확인)

## 스코프 확인
- MVP 포함 여부 (docs/roadmap.html, scope-reviewer 관점 필요)

## 미확정 — 상의 필요
-
```

**점검 규칙:**
- 퀘스트가 "필수 강제"처럼 설계되지 않았는지 CLAUDE.md 핵심 디자인 철학 3번(옵트인)과 대조한다 — 애매하면 `design-check` 스킬로 먼저 점검한다.
- 보스/던전 관련 퀘스트는 `docs/systems/adventure-combat.html`의 "최적 해법형" 원칙(자물쇠-열쇠 금지)을 위반하지 않는지 확인한다.
- 완성된 퀘스트 목록은 `docs/quests/README.md`(또는 `.html`)에 표로 인덱싱한다 (없으면 새로 생성).

**미확정 — 산출물 형식:** `docs/`가 html로 전환됨에 따라 위 템플릿을 `.md`로 쓸지 `.html`로 쓸지 아직 결정되지 않았다. 결정 전까지는 사용자에게 먼저 확인한다.
