# 주제 딥다이브 브레인스토밍 스킬

**목적:** "미확정 — 상의 필요" 항목 중 하나를 골라, model recall이 아닌 실제 웹 리서치로 근거를 확보하고, 관련 전문 에이전트들의 실제 입장 차이(억지 합의 아님)를 드러낸 뒤 구체적 기획안으로 종합한다. 진행 방식은 2026-07-03 "마을 개방 구조" 세션에서 실제로 검증됨 (docs/meetings/2026-07-03-village-opening-structure.html 참고).

**활성화:** "이 주제로 에이전트들 브레인스토밍해줘", "레퍼런스 조사해서 기획안 만들어줘", "~에 대해 에이전트들 의견 모아줘" 등.

## 절차

1. **주제 확정**: 사용자가 특정 미확정 항목을 지정하지 않았으면 `resolve-open-questions`로 먼저 안건 목록을 보여주고 하나를 고르게 한다. 한 세션에 여러 주제를 한꺼번에 딥다이브하지 않는다 (품질 저하).
2. **리서치 브리핑 작성 (메인 세션이 직접 수행, 반드시 실제 WebSearch/WebFetch)**:
   - 주제와 관련된 실제 게임/사례를 웹에서 조사한다. 모델 기억에 의존한 "~일 것이다" 서술은 금지 — 반드시 URL과 함께 인용.
   - WebFetch로 원문을 확인한 근거는 "신뢰도 높음", WebSearch 스니펫만으로 종합한 근거는 "신뢰도 낮음/참고 수준"으로 명시해 구분한다 (근거 강도를 부풀리지 않는다).
   - 리서치는 **한 번만** 수행해서 모든 에이전트에게 동일하게 공유한다 — 에이전트마다 각자 웹 검색을 시키지 않는다 (중복/불일치 방지, technical-advisor 외 에이전트는 기본적으로 웹 도구가 없다).
3. **관련 에이전트 병렬 소집**: 주제와 관련된 에이전트만 골라 Agent 도구로 **한 메시지에서 병렬로** 스폰한다 (보통 game-designer + 주제별 전문 에이전트 1~3개 + scope-reviewer는 스코프 영향이 있으면 항상 포함). 각 에이전트에게 동일한 리서치 브리핑을 프롬프트에 포함하고 다음을 명시적으로 요구한다:
   - 명확한 하나의 추천(양자택일/다자택일 중 하나를 실제로 고르게 함, 애매하게 답변 금지)
   - 이 프로젝트 확정 철학(CLAUDE.md)과의 부합/충돌 여부
   - **다른 관점과 충돌할 만한 지점을 스스로 명시** (합의된 척 하지 말 것 — 이게 빠지면 브레인스토밍이 아니라 그냥 평균값이 된다)
   - 200~300단어 제한, "제안일 뿐 확정 아님" 명시
4. **종합 (메인 세션이 수행)**:
   - 각 에이전트 입장을 요약한 표를 만든다.
   - **실제 충돌 지점을 숨기지 않고 별도 섹션으로 드러낸다.** 종합안이 마찰 없는 합의처럼 보이면 실패한 것이다 — 진짜 브레인스토밍이라면 보통 "철학/창작 쪽"과 "scope-reviewer(비용)" 사이에 진짜 이견이 남는다.
   - 충돌을 조정하는 실마리(예: 기술적으로 정책과 구현을 분리해 나중에 되돌릴 수 있게 만드는 식의 제안)가 있으면 그것을 축으로 종합 추천을 구성한다.
   - 이 단계까지는 **파일에 아무것도 쓰지 않는다** — 대화로만 제시하고 사용자 확인을 기다린다.
5. **사용자 확인**: 종합안을 사용자에게 제시하고 확정 여부를 묻는다. 사용자가 바꾸고 싶어하면 몇 라운드든 대화로 조정한다 — 이 스킬 자체가 최종 확정 권한을 갖지 않는다.
6. **확정 시 문서 반영**:
   - `docs/meetings/<날짜>-<주제-슬러그>.html`에 회의록 작성: 안건, 참가 에이전트, 리서치 브리핑(출처/신뢰도 포함), 에이전트별 입장 표, 충돌 지점, 종합 추천, 사용자 최종 결정.
   - `docs/meetings/README.html` 목록에 행 추가, `docs/index.html`에도 링크 추가.
   - 결정 내용을 실제 대상 문서(예: `docs/risks.html`, `docs/systems/villages.html`)의 "미확정 — 상의 필요" 섹션에서 제거하고 "확정된 사항" 섹션으로 옮기며, 회의록 링크를 건다.
   - `docs/pre-production-checklist.html`에 관련 ID가 있으면 상태 갱신 + "변경 이력"에 append.
   - 부분적으로만 해결된 하위 항목은 "완료"로 과장하지 말고 "진행중"으로 정직하게 표기한다.

## 이 스킬이 아닌 것
- 여러 주제를 한 번에 훑어 목록화하는 것 → `resolve-open-questions`
- 사용자와의 열린 논의를 여러 라운드 이어가는 것(에이전트 스폰 없이) → `open-question-facilitator` 에이전트
- 이 스킬은 "리서치 + 다중 전문가 병렬 의견 + 명시적 충돌 + 종합"이 핵심이며, 이 중 하나라도 생략하면(특히 리서치나 충돌 표면화) 이 스킬을 쓴 게 아니다.
